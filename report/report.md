# Тема
Сравнительный анализ производительности реализаций API Gateway в микросервисной архитектуре
# Введение
Увеличение сетевой нагрузки, объемов данных и требований к надежности и масштабируемости современного программного обеспечения привело к росту популярности распределенных систем и широкому внедрению микросервисной архитектуры. Любая подобная система сталкивается с необходимостью наличия единого шлюза, который обеспечивает маршрутизацию, аутентификацию и другие функциональные задачи. Такой подход в микросервисной архитектуре описывается паттерном API Gateway. На сегодняшний день существуют как готовые реализации этого паттерна с открытым исходным кодом, так и фреймворки, позволяющие добиться более гибкого и подходящего поведения API Gateway.
Выбор конкретной реализации API Gateway на практике часто определяется не только функциональными возможностями, но и характеристиками производительности: пропускной способностью, задержкой, устойчивостью под нагрузкой. API Gateway участвует в обработке каждого запроса, поэтому даже небольшие накладные расходы на этом этапе могут существенно повлиять на итоговую производительность всей системы. Это делает сравнение шлюзов актуальным не только с точки зрения удобства разработки и сопровождения, но и с позиции эксплуатационных затрат и соблюдения требований по SLO/SLА.
Особый интерес представляет сопоставление решений в экосистеме Spring, которая предоставляет два разных подхода к реализации собственного API gateway. Spring Cloud Gateway позволяет описать собственный шлюз с использованием реактивных подходов и возможностью расширения функционала через механизм фильтров. Spring Cloud Gateway MVC, напротив, опирается на более привычный Spring-разработчику сервлетный стек, отказываясь от преимуществ реактивного программирования, но делая акцент на поддержке виртуальных потоков из Java 21. Важно учесть в сравнении и готовые решения, получившие популярность благодаря своей простоте и эффективности. Nginx широко применяется как высокопроизводительный обратный прокси и балансировщик. При этом сопоставление таких решений требует аккуратно заданной методики: единых сценариев маршрутизации, сопоставимых политик таймаутов и повторов, одинаковой конфигурации.
Целью данной работы является сравнительный анализ производительности API Gateway в роли обратного прокси на примере трех реализаций: nginx, Spring Cloud Gateway и Spring Cloud Gateway MVC. В рамках исследования планируется определить набор репрезентативных сценариев использования (простейшая проксирующая маршрутизация, цепочки фильтров, аутентификация/авторизация, ограничение частоты запросов), сформировать воспроизводимую методику нагрузочного тестирования и провести измерения ключевых метрик: задержки (p50/p95/p99), пропускной способности, процента ошибок, устойчивости при росте конкурентности, а также потребления CPU/памяти. Результаты сравнения предполагается интерпретировать с учетом особенностей архитектуры и модели выполнения каждого решения, что позволит сформулировать практические рекомендации по выбору API Gateway в зависимости от требований к производительности и окружения эксплуатации.
